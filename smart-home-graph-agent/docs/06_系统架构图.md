### 谁负责什么？

```
┌─────────────────────────────────────────────────────────────────┐
│                    责任分工矩阵                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  任务                      │ 谁负责       │ 为什么是它          │
│  ─────────────────────────┼──────────────┼──────────────────── │
│  理解"弄得舒服一点"       │ LLM (OpenAI) │ 需要语义理解        │
│  决定先解析再检索再规划   │ LangGraph    │ 这是流程编排        │
│  构造提示词发给LLM        │ LangChain    │ 这是它的核心能力    │
│  找到客厅有哪些设备       │ GraphRAG     │ 需要图遍历检索      │
│  存储设备和能力的关系     │ Neo4j        │ 原生图存储          │
│  解析LLM的JSON输出        │ LangChain    │ Output Parser       │
│  决定是否需要追问用户     │ LangGraph    │ 条件路由            │
│  验证计划是否可行         │ Agent节点    │ 业务逻辑验证        │
│  生成友好的用户回复       │ LLM (OpenAI) │ 需要自然语言生成    │
│  管理整个处理过程的状态   │ LangGraph    │ 状态管理            │
│                                                                 │
│  关键理解:                                                      │
│  • LLM负责"思考" (理解和生成)                                  │
│  • LangGraph负责"流程" (何时做什么)                             │
│  • LangChain负责"沟通" (与LLM交互)                             │
│  • GraphRAG负责"找" (从图谱检索)                               │
│  • Neo4j负责"存" (持久化知识)                                  │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 6.3 一次请求的完整生命周期

### 端到端数据流

```
┌═════════════════════════════════════════════════════════════════┐
║  完整请求生命周期: "Make the living room cozy for movie night"  ║
╠═════════════════════════════════════════════════════════════════╣
║                                                                 ║
║  ❶ 用户输入                                                    ║
║  ┌─────────────────────────────────────────────────────────┐   ║

║  └─────────────────────────────────────────────────────────┘   ║
║       │                                                         ║
║       │  Streamlit / CLI 接收输入                               ║
║       │  调用 agent.run(user_input)                             ║
║       │                                                         ║
║  ─────┼──────────── LangGraph 开始编排 ──────────────────────  ║
║       │                                                         ║
║       ▼                                                         ║
║  ❷ 节点: parse_intent  [LangGraph调度 → LangChain调用LLM]      ║
║  ┌─────────────────────────────────────────────────────────┐   ║
║  │                                                          │   ║
║  │  LangChain:                                              │   ║
║  │    prompt = INTENT_PARSER_PROMPT.format(user_input=...)  │   ║
║  │    response = llm.invoke(prompt)                         │   ║
║  │    intent = extract_json(response.content)               │   ║
║  │                                                          │   ║
║  │  结果写入State:                                          │   ║
║  │  parsed_intent = {                                       │   ║
║  │      "goal": "create cozy movie atmosphere",             │   ║
║  │      "rooms": ["living room"],                           │   ║
║  │      "mood": "cozy",                                     │   ║
║  │      "scene_hint": "movie night"                         │   ║
║  │  }                                                       │   ║
║  └─────────────────────────────────────────────────────────┘   ║
║       │                                                         ║
║       │  LangGraph: 普通边 → 下一节点                           ║
║       ▼                                                         ║
║  ❸ 节点: retrieve_context  [LangGraph调度 → GraphRAG检索]      ║
║  ┌─────────────────────────────────────────────────────────┐   ║
║  │                                                          │   ║
║  │  GraphRAG策略选择:                                       │   ║
║  │    rooms=["living room"] → 使用ROOM_CONTEXT策略          │   ║
║  │    scene_hint="movie night" → 使用SCENE_LOOKUP策略       │   ║
║  │                                                          │   ║
║  │  Cypher查询1 (Room):                                     │   ║
║  │    MATCH (r:Room)-[:CONTAINS]->(d:Device)                │   ║
║  │    WHERE r.name CONTAINS "Living"                        │   ║
║  │    OPTIONAL MATCH (d)-[:HAS_CAPABILITY]->(c:Capability)  │   ║
║  │    ...                                                   │   ║
║  │                                                          │   ║
║  │  Neo4j返回:                                              │   ║
║  │    Living Room → [Smart TV, Ceiling Light, Floor Lamp,   │   ║
║  │                   Smart Speaker, Window Blinds,          │   ║
║  │                   Thermostat]                             │   ║
║  │    各设备能力: [power, dim, color, volume, ...]           │   ║
║  │                                                          │   ║
║  │  Cypher查询2 (Scene):                                    │   ║
║  │    MATCH (s:Scene {name: "Movie Night"})...              │   ║
║  │                                                          │   ║
║  │  Neo4j返回:                                              │   ║
║  │    Movie Night: [TV→power on, Light→dim 20%, ...]        │   ║
║  │                                                          │   ║
║  │  格式化为Markdown写入State:                              │   ║
║  │  retrieved_context = "# Room: Living Room\n## Devices..."│   ║
║  │                                                          │   ║
║  └─────────────────────────────────────────────────────────┘   ║
║       │                                                         ║
║       ▼                                                         ║
║  ❹ 节点: check_sufficiency  [LangGraph调度 → 逻辑判断]         ║
║  ┌─────────────────────────────────────────────────────────┐   ║
║  │                                                          │   ║
║  │  检查: context是否包含设备信息? ✓                        │   ║
║  │  检查: rooms是否已指定? ✓                                │   ║
║  │  结果: needs_clarification = False                       │   ║
║  │                                                          │   ║
║  └─────────────────────────────────────────────────────────┘   ║
║       │                                                         ║
║       │  LangGraph: 条件边 → route_after_check()               ║
║       │  返回 "generate_plan"                                   ║
║       ▼                                                         ║
║  ❺ 节点: generate_plan  [LangGraph调度 → LangChain调用LLM]     ║
║  ┌─────────────────────────────────────────────────────────┐   ║
║  │                                                          │   ║
║  │  LangChain:                                              │   ║
║  │    prompt = ACTION_PLAN_PROMPT.format(                    │   ║
║  │        parsed_intent=intent_json,                        │   ║
║  │        retrieved_context=context_markdown                │   ║

║  │    response = llm.invoke(prompt)                         │   ║
║  │                                                          │   ║
```

### 调用次数统计

```
┌─────────────────────────────────────────────────────────────────┐
│                   一次请求的资源消耗                             │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌───────────────────────┬────────┬───────────────────────────┐│
│  │ 组件                  │ 次数   │ 说明                      ││
│  ├───────────────────────┼────────┼───────────────────────────┤│
│  │ LLM调用 (OpenAI API) │  3次   │ 解析 + 规划 + 响应        ││
│  │ Neo4j查询             │  2次   │ Room查询 + Scene查询      ││
│  │ LangGraph节点执行     │  7次   │ 6个处理节点 + 1个计数器   ││
│  │ 条件路由判断          │  2次   │ sufficiency + validation  ││
│  └───────────────────────┴────────┴───────────────────────────┘│
│                                                                 │
│  注意:                                                          │
│  • 如果验证失败，会多1次LLM调用 + 1次Neo4j查询                 │
│  • 如果需要澄清，只需1次LLM调用，流程提前结束                  │
│  • 整体设计目标: 最少3次LLM调用完成一次请求                    │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 6.5 设计权衡与常见误区

### 常见误区

```
┌─────────────────────────────────────────────────────────────────┐
│                      常见误区                                    │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  误区1: "LLM可以做一切"                                        │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ ❌ 让LLM直接查询数据库                                   │   │
│  │ ❌ 让LLM记住所有设备信息                                 │   │
│  │ ❌ 不用检索，直接让LLM回答                               │   │
│  │                                                          │   │
│  │ ✅ 正确做法:                                              │   │
│  │    LLM负责理解和推理                                     │   │
│  │    GraphRAG负责精确检索                                  │   │
│  │    LangGraph负责流程编排                                 │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  误区2: "知识图谱只适合大规模数据"                              │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ ❌ "我只有50个节点，不需要图数据库"                       │   │
│  │                                                          │   │
│  │ ✅ 正确理解:                                              │   │
│  │    图谱的价值在于关系的表达能力，而非数据量                │   │
│  │    50个节点的图可能有200+关系                             │   │
│  │    这些关系用传统数据库需要很多JOIN                       │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  误区3: "GraphRAG可以替代传统RAG"                               │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ ❌ 所有场景都用GraphRAG                                   │   │
│  │                                                          │   │
│  │ ✅ 正确理解:                                              │   │
│  │    • 结构化知识 → GraphRAG                               │   │
│  │    • 非结构化文本 → 传统RAG                              │   │
│  │    • 混合场景 → Hybrid RAG (两者结合)                    │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  误区4: "Agent就是多调几次LLM"                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ ❌ 简单循环调用LLM就是Agent                               │   │
│  │                                                          │   │
│  │ ✅ 正确理解:                                              │   │
│  │    Agent = 状态管理 + 工具使用 + 条件推理 + 错误处理     │   │
│  │    LangGraph提供了这些能力的框架支持                     │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  误区5: "LangChain和LangGraph是同一个东西"                      │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ ❌ LangGraph只是LangChain的一部分                        │   │
│  │                                                          │   │
│  │ ✅ 正确理解:                                              │   │
│  │    • LangChain: LLM交互框架 (Prompt → LLM → Parse)      │   │
│  │    • LangGraph: 工作流框架 (State → Node → Edge → END)  │   │
│  │    • LangGraph使用LangChain的组件，但有独立的API         │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 设计权衡

```
┌─────────────────────────────────────────────────────────────────┐
│                      设计权衡                                    │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  权衡1: LLM调用次数 vs 输出质量                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 少次调用: 一次LLM完成所有(快但可能质量差)               │   │
│  │ 多次调用: 分步处理(慢但质量高、可调试)                   │   │
│  │                                                          │   │
│  │ 本项目选择: 3次调用 (解析+规划+响应) ← 平衡点           │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  权衡2: 预定义查询 vs 动态Cypher生成                            │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 预定义: 安全、可预测、但灵活性低                         │   │
│  │ 动态生成: 灵活、但可能出错、有安全风险                   │   │
│  │                                                          │   │
│  │ 本项目选择: 预定义模板 ← 教学项目优先安全可控            │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  权衡3: 简单验证 vs LLM验证                                     │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │ 规则验证: 快速、确定性、但覆盖面窄                       │   │
│  │ LLM验证: 灵活、覆盖面广、但慢且不确定                   │   │
│  │                                                          │   │
│  │ 本项目选择: 规则验证 ← 减少LLM调用和成本                │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```
